## Notes

- Current stack remains FastAPI/Pydantic, repository-backed Mongo/in-memory database, React frontend, and Docker deployment.
- Collection names are not destructively renamed for blueprint alignment.
- Segment-scoped request service, pet, and special item classes are now populated by Phase 34 normalization.
- Trip dossier classes now support manual trip creation, request-to-trip conversion, request linking, copied request child records, and trip timelines while preserving request independence.
- Reference Data is platform-owned global master data; agency suggestions improve it only after platform review.
- Reference Data now includes an explicit consumer map and health/action-required layer so platform owners can see which workflows consume each domain and why records need attention.
- Reference Data is intentionally separate from Airline Intelligence policy rules; it supplies lookup values and service catalogue metadata, not airline-specific acceptance decisions.
- Phase 36.0 adds the shared Rules & Services foundation: platform owns global airline rules and exception records, while agencies consume/evaluate them in request and trip workflows.
- Phase 36.1 adds additive offer workspaces and internal comparison matrices without destructively renaming the legacy Phase 4 offer collections.
- Phase 36.2 adds accepted-offer snapshots and booking readiness packages; these are readiness-only records and do not create live bookings, PNRs, tickets, EMDs, invoices, payments, or supplier actions.
- Phase 36.2.5 makes Service Catalogue editable and operational while preserving compatibility with existing service codes and snapshots.
- Phase 36.3 adds booking workspaces and draft manual PNR mirrors from readiness packages; provider execution, ticketing, EMD issuance, invoices, and payments remain outside that foundation.
- Phase 36.4 adds internal ticket and EMD mirrors, coupon records, lifecycle events, service catalogue mapping preservation, and booking-record ticket/EMD readiness summaries; live ticketing, EMD issuance, provider calls, settlement, invoices, and payments remain outside this foundation.
- Phase 36.4.5 adds supplementary blueprint sync foundations and documents canonical route policy; `/agent` and `/admin` roots are intentionally rejected while `/platform` and `/agency` remain canonical.
- Phase 36.4.6 adds standalone manual booking/ticket/EMD mirrors, booking import drafts, existing-trip change operations, and ticket/EMD exchange operation mirrors; provider execution, real exchange/reissue/refund/void, payments, invoices, and accounting remain outside this foundation.
- Phase 36.5 adds document render jobs, packages, share records, platform default templates, and a normalized context builder over offer/trip/booking/ticket/EMD/import/change/service records; live delivery, e-signature, payments, invoices/accounting, settlement, and provider execution remain outside this foundation.
- Phase 36.6 adds governed GDS parser profiles, versions, runs, parsed entities, corrections, training samples, evaluations, booking-import draft integration, and parser-run document context; live GDS/NDC/provider execution, external AI parsing, automatic imports, and full host grammar coverage remain outside this foundation.
- Phase 36.7 adds airline policy sources, detected sections, extraction runs, rule/price/communication/EMD/exception candidates, review corrections, and approved policy knowledge records; external AI extraction, scraping, live provider calls, automatic promotion, and full comparison/SSR/OSI/EMD engines remain outside this foundation.
- Phase 36.8 adds canonical special/ancillary service taxonomy domains, families, variants, aliases, mapping rules, applicability dimensions, outcome types, policy candidate taxonomy links, and taxonomy review corrections; it remains the canonical meaning layer and does not itself execute SSR/OSI communication, EMD/payment mechanics, pricing matrices, comparison matrices, scraping, external AI mapping, live provider execution, or agency auto-promotion.
- Phase 36.9 adds SSR/OSI communication mechanics and separate EMD/RFIC/RFISC payment mechanics mapping; live SSR/OSI transmission, live provider execution, ticket/EMD issuance, payments, settlement, scraping, external AI mapping, and agency auto-promotion remain outside this foundation.
- Phase 37.0 adds ancillary pricing rules/components/applicability/matrices, expanded service exception rules, quote scenarios/results, and candidate pricing links; price estimates are not invoices/payments/accounting, EMD references do not issue EMDs, exception rules are advisory/governance metadata, and provider execution remains disabled.
- Phase 37.1 adds airline policy comparison profiles/snapshots/rows, service advisor scenarios/results, and saved views; the advisor is metadata-only, complexity scores are not automatic airline recommendations, and provider execution, EMD issuance, payments, invoices, accounting, and settlement remain disabled.
- Phase 37.2 adds offer policy advisor contexts, offer-linked airline rows, warnings, manual decision notes, and saved snapshots; advisor context is metadata-only and never auto-selects airlines, changes offer pricing, books, issues, charges, invoices, settles, calls providers, scrapes, or calls external AI.
- SSR/OSI output is deterministic preview text for staff review; it is not ticketing, booking execution, or an AI reasoning engine.
- Policy learning now starts with governed source ingestion and reviewed approved knowledge records; later promotion into canonical rules/services engines should remain explicit and audited.
- Field profiles are agency-owned display configuration over platform-owned canonical fields; custom agency questions must remain in `agency_custom_fields`.
- GDS parser runs and airline policy sources/extraction runs link to review summary document contexts.
