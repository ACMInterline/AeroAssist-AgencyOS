# Phase 14 Document Delivery Hardening

## Goal

Harden Phase 13 document export and email delivery without adding public links, automatic sending, client-triggered delivery, payment links, signatures, uploads, marketing campaigns, airline integrations, refund/payment execution, or fiscal invoice compliance claims.

Document delivery remains staff-controlled and snapshot-based.

## Storage Behavior

New printable HTML exports are stored through `backend/services/file_storage_service.py`.

- Default storage root: `.local/document_exports`
- Configurable with: `DOCUMENT_EXPORT_STORAGE_DIR`
- Storage keys are generated by the server and path traversal is rejected.
- Export records store `storage_mode=file_path`, `storage_bucket=local`, `storage_key`, `file_size_bytes`, and `checksum_sha256`.
- Downloads verify checksum when present.
- Existing `inline_base64` exports remain downloadable for backward compatibility.

## PDF Behavior

Real PDF rendering remains unavailable in this installation.

The current backend dependency set does not include a verified HTML-to-PDF renderer such as ReportLab, WeasyPrint, or xhtml2pdf, and no Poppler-style visual QA path is installed. PDF export requests still fail honestly with a friendly error and do not create fake downloadable PDFs.

## Email Settings And Secret Behavior

Email modes remain:

- `disabled`
- `dev_console`
- `smtp`

The API does not store or return raw SMTP passwords. SMTP configuration requires a password secret reference or configured-secret flag. Because no production secret resolver exists yet, SMTP validation and send attempts fail gracefully with a clear secret-resolution message.

Added:

- `POST /api/agencies/{agency_id}/email-settings/validate`

The validation endpoint checks configuration completeness and records validation state; it does not send a test email.

## Delivery Attempts And Retry Behavior

Each staff-controlled send or retry creates a `DocumentDeliveryAttempt`.

- `dev_console` attempts are marked sent and update the delivery to sent.
- `disabled` attempts are recorded as failed.
- incomplete or unresolved SMTP attempts are recorded as failed.
- failed deliveries become retryable until `max_attempts` is reached.
- retry is staff-controlled through:
  - `POST /api/agencies/{agency_id}/document-deliveries/{delivery_id}/retry`
- attempts are readable by staff through:
  - `GET /api/agencies/{agency_id}/document-deliveries/{delivery_id}/attempts`

No background worker, automatic retry, or automatic send-on-render behavior exists.

## Portal Visibility

Portal document downloads remain read-only.

Portal users can list and download only generated, client-visible exports for rendered documents that belong to the authenticated portal client. Portal export projections omit storage keys, bucket paths, checksums, delivery records, attempts, internal notes, and snapshots.

Repeated seed runs do not duplicate the demo export, delivery, settings, or attempt.

## Remaining Production Risks

- No real PDF renderer or visual PDF QA path is installed.
- Local filesystem storage is a foundation, not a full production object-storage lifecycle.
- No background queue, retry worker, provider webhooks, bounce handling, or delivery monitoring exists.
- No production secret resolver exists for SMTP credentials.
- No public links, expiring URLs, client-triggered sends, marketing/bulk send, payment links, e-signatures, uploads, or fiscal invoice compliance output exists.
- Formal migrations and broader automated authorization tests remain future hardening work.

## Exact Next Recommended Phase

Phase 15 should be Production PDF Rendering And Delivery Infrastructure: select and verify a real PDF renderer with visual QA, add production object storage/retention lifecycle, integrate a secret resolver, and add background queue/retry/provider webhook handling before considering public links or automated delivery.
